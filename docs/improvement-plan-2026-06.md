# Buysell — Improvement Plan (Audit 2026-06-19)

Generated by a multi-agent audit: 17 finders (12 subsystem + 5 cross-cutting) produced 178 candidate findings; each was adversarially verified against the source. **139 confirmed real.**

- **Severity:** 0 critical · 23 high · 42 medium · 74 low
- **Category:** security 26 · concurrency 21 · data 27 · bug 22 · feature 23 · performance 18 · cleanup 2
- **Effort:** S = <2h · M = ~half-day · L = multi-day

## Executive summary

This is an ambitious, feature-rich arbitrage system whose architecture is sound but whose money- and marketplace-touching paths are dangerously under-hardened — the gap between "what the code advertises as a safety guarantee" and "what it actually enforces" is the dominant risk. Three classes of P0 defect compound each other: every spend/listing guardrail (daily purchase cap, daily listing cap, action execution) is implemented as check-then-act outside a transaction with no atomic claim, so concurrency or a double-click can spend real money twice and publish duplicate live listings; the advertised kill-switches (safeMode, the DRAFT/ASSISTED human gate, the autopilot evidence gate, max-price) are not actually enforced before irreversible side effects; and the auth perimeter is bypassable (X-Forwarded-For-spoofable rate limiting on the only credential, plaintext shared secret sent every request, replayable HMAC). Separately, the headline financial feature is hollow — ProfitLedgerEntry is never written, so realized P/L is permanently $0 — and the "learning" loop only collects data, never reading it back. Money-math is systematically optimistic (ROI denominator and fee base both omit real costs), and there is no fresh price re-check at purchase time, no automated repricing, no order-sync scheduler, and no off-dashboard alerting. None of the highest-risk paths (executor, orders, credentials) have tests. The good news: most P0 items are small, surgical fixes (atomic updateMany claims, moving checks inside transactions, honoring existing flags), so a focused hardening sprint can de-risk the system quickly before tackling the larger correctness and product gaps.

## Themes & findings

### [P0] Concurrency & double-execution safety (real money/listing side effects)

_Every money- and marketplace-mutating path is check-then-act with no atomic claim and external API calls happen before the DB transaction. A double-click, retry, two local-agent workers, or overlapping scheduler ticks can spend real money twice, publish duplicate live eBay listings, and blow past the configured daily spend/listing caps. This is the single highest-impact risk class in the codebase._

## New feature proposals

### Write the realized P/L ledger  _(value-tier effort: M)_

**Why:** The dashboard's headline financial metric is permanently $0 and the profit-ledger CSV is always empty because ProfitLedgerEntry is never written and ProductFamily.realizedProfit is never incremented. For a money-handling tool this makes the primary report misleading and hides whether the business is actually profitable.

**What:** On purchase recording (recordAmazonPurchaseRows) or a reconciliation job, create a linked ProfitLedgerEntry (revenue, sourceCost, marketplaceFees, shippingCost, refunds, netProfit) and increment ProductFamily.realizedProfit in the same idempotent transaction so realized P/L and the export reflect real outcomes.

### Off-dashboard alerting/notifications  _(value-tier effort: L)_

**Why:** Alerting is entirely pull-based (only when a browser polls /api/alerts). The highest-value arbitrage events — a profitable LIST opportunity, an Amazon price spike PAUSE, a pending BUY before the buyer waits, a FAILED automation run, near-exhausted Keepa tokens — never reach a human unless a tab is open.

**What:** Add a pluggable notification service (webhook + email to start) fired from existing event seams (PAUSE created, BUY created, run FAILED, opportunity created), with per-event thresholds in Settings mirroring the dashboardAlerts categories.

### Fresh price/availability re-check at purchase time  _(value-tier effort: M)_

**Why:** BUY actions use the maxPrice captured at listing time; by the time an eBay item sells the Amazon source price may have risen or gone out of stock, locking in a loss. Docs claim this re-check exists but it is not implemented.

**What:** In createOrderAndBuyAction call getAmazonProductByAsin for live buy-box price/availability, recompute landed-cost profit, set a fresh maxPrice, and route to MANUAL_REVIEW (or skip BUY) when price exceeds threshold or stock is gone; record a PriceObservation.

### Automated repricing & auto-resume engine  _(value-tier effort: L)_

**Why:** The monitor only one-way PAUSEs on raw Amazon cost increase. It never raises eBay price to preserve margin, never lowers price to stay competitive, and never resumes a PAUSED listing when Amazon price recovers. REPRICE exists in the schema/executor but is only ever created manually. Dynamic repricing is a core revenue feature for arbitrage.

**What:** Build an engine that emits REPRICE and resume ActionItems gated by RuleConfig thresholds and the existing approval/daily-limit pipeline; base PAUSE/REPRICE on profitability (eBay price vs current Amazon cost minus fees/buffers) rather than any raw uptick, include PAUSED listings in the scan, and re-sample eBay comps.

### Out-of-stock / sourcing-risk pause and inventory sync  _(value-tier effort: L)_

**Why:** The monitor ignores Amazon availabilityStatus and only reacts to price increases; an item going out of stock (or losing its buy box) leaves the eBay listing ACTIVE and unfulfillable. SourceInventoryRecord is created once at quantityOnHand:0 and never updated.

**What:** Add an availability-sync pass that refreshes SourceInventoryRecord from Keepa stock status, treats sustained NO_PRICE / out-of-stock as a sourcing-risk PAUSE, reserves quantity on eBay order creation, and tightens the keepaClient availabilityAmazon mapping (e.g. -1 -> OUT_OF_STOCK).

### Close the learning loop (read feedback back into scoring + backtesting)  _(value-tier effort: L)_

**Why:** The system writes a rich learning dataset (OpportunityFeedback weights, ProductFamily decision counters, PriceObservation history) but the scorers and discoveryPolicy never read any of it, so repeated rejections and manual overrides never affect future rankings, and there is no way to validate score/threshold values against realized profit.

**What:** Feed ProductFamily history (reject/list ratio, override weights) into the scorers as a damp/boost factor, and add a backtesting harness that re-scores historical candidates against realized ProfitLedgerEntry outcomes to tune minimumOpportunityScore/ROI/match-confidence thresholds.

### Automatic eBay order-sync scheduler  _(value-tier effort: M)_

**Why:** Order intake only happens when an operator clicks 'Sync' — there is no scheduler for it (unlike price monitoring, discovery, and comparison), adding latency between an eBay sale and the Amazon purchase and risking buyer-wait SLA breaches and stale source prices, with no recovery for sales missed during downtime.

**What:** Add a DB-locked order-sync scheduler mirroring amazonPriceMonitorScheduler (reusing withSchedulerLock), driven by a new RuleConfig interval with a lookback overlap to survive restarts, surfacing unknown-listing skips as alerts.

### Verification freshness TTL + re-verify at execution  _(value-tier effort: M)_

**Why:** A PASSED verification has no expiry and execution never re-checks economics against current prices, so a LIST/BUY approved hours/days later can act on a stale price snapshot — a direct path to negative-margin listings in a volatile market.

**What:** Add a configurable verification TTL and re-run calculateProfit against current prices in executeAction LIST/BUY, rejecting or forcing re-verification when checkedAt is older than the TTL.

### Retry/backoff + dead-letter for failed automation runs  _(value-tier effort: M)_

**Why:** A FAILED automation run does not block re-polling and the action stays APPROVED, so any reliably-failing action is retried every poll cycle forever, repeatedly creating FAILED runs and burning operator launches / external resources, while never surfacing as a terminal 'needs manual intervention' state.

**What:** Add an attempt counter + backoff window on AutomationRun, exclude recently-FAILED operator-mode actions from polling, and after N failures transition the ActionItem to ERROR with the last failure reason surfaced in a dashboard triage list.

### Reuse persisted Keepa/SerpApi payloads on force re-compare  _(value-tier effort: M)_

**Why:** Re-running a comparison after tweaking thresholds (a pure local re-scoring operation) always re-calls Keepa/SerpApi from scratch, re-purchasing identical market data even though raw payloads are already persisted on the candidate.

**What:** On force re-compare of REJECTED/MANUAL_REVIEW candidates, re-score locally from the persisted scoreBreakdown/raw payloads when fresh, and add an explicit 'refresh prices' flag to force new fetches.
